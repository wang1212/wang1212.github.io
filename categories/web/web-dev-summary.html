Web 前端开发总结

Web    前端    流程    规范

Web 开发涉及到后台与前端的分工，前端使用 HTML、CSS、JS 进行结构、样式、交互的分类开发，前端开发没有一款像后台开发那样功能强大的开发工具，自动化工具配置也较为复杂，代码文件多带来的后果就是效率的降低。规范化前端开发，对于提升项目开发效率有极大的意义。

发表于 2017-9-20  分类 Web

前端工具

前端开发这边有很多著名的自动化开发工具，像 grunt、gulp 等等，功能强大也意味着配置复杂。像我们许多新手，则喜欢轻装上阵，在这里推荐一款极简风格而且功能强大的编辑器 Sublime Text ，代码自动完成、格式化等等功能会给你一个惊喜。

Sublime Text : http://www.sublimetext.com/

除过这些项目构建的开发工具以外，前端也极为注重性能，所以代码压缩，代码重用性，资源图片的压缩等等相关的工具也极为重要，这些可以从互联上进行了解学习，例如 CSS 预处理器 Less、Sass、Compass。一定要多了解前端开发工具，即便不用也有助于提升自己的开发效率。

开发流程

这里所说的开发流程则侧重于实际代码开发的流程，项目规划完成后，要进行实际化的代码开发，此时不能是随意的开始写一个页面。一般应先考虑以下几点：

  • 项目页面兼容性需求
  • 项目页面自适应需求
  • 框架选择
  • 页面架构

页面兼容性与自适应需求是应首先要考虑的问题，因为前端开发的难点在于浏览器之间的统一，而要做到大多数浏览器统一的显示和交互效果，兼容性是躲不过的。而兼容性又分为 HTML5 新标签、CSS3 新样式、JS 解析兼容几大类别，其中 CSS 与 JS 兼容是要着重考虑的问题，毕竟 Win7 默认自带还是 IE8，这是我们开发者的痛点。其次，多数项目需求可能既要满足客户端用户也要满足移动端用户，自适应问题接踵而来，可以解决自适应的 CSS 框架最常见的就是 Bootstrap 了，了解一下其实现自适应的原理是很有必要的；还有 jQuery 类库可以更方便的操作 DOM，并解决一些兼容问题。

html5shiv.js : https://github.com/aFarkas/html5shiv/
selectivizr.js : https://github.com/keithclark/selectivizr
Respond.js : https://github.com/scottjehl/Respond

以上三个工具都是用来解决 IE6-8 兼容性问题，其中 html5shiv.js 用来提供对 html5 新标签（例如 footer、nav）的支持；selectivizr.js 用来提供对 CSS3 新选择器（例如 :nth-child、:last-child）的支持；Respond.js 用来提供对 CSS3 @media媒体查询机制（min-width/max-width）的支持。

说完兼容性和自适应需求才可以谈谈到底选择什么框架，前端目前比较火的框架像 React 、 Vue.js 这些属于显示层的，我们用起来门槛较低，一般官方各版本都会说明浏览器的兼容级别，这些界面层的框架其实最终目的是帮助我们减少手写 重复的 DOM 结构的工作，将类似 DOM 结构作为一个个组件，实现重用，其次像事件、数据交互这些也更为好管理一些。而其它的像 Aangular.JS 这种就是比较重的 MVVM 框架了，类似于传统的 Web 开发中的 MVC 模式，将更多后台要做的事情拿到前台来做，提高应用性能和体验，门槛相对较高。

通常项目规划完成，拿到设计图之后，对页面要进行架构设计，写一个页面不是从上写到下，而是先写好页面的整体框架，整体框架实现之后，再去填充每一部分内部的结构。

CSS 规范化

在着手开始写页面的时候，首先要将各种浏览器的 CSS 默认样式的差异消除到最小，也就是 CSS 样式的规范化。许多公司有自己的初始化代码，但大多都大同小异， Normalize.css 是我们开发者一个不错的选择，它还能修复一些浏览器 BUG，省事省心。当然，也有一种叫做 CSS Reset 的方式来强行将所有元素的默认样式统一，也就相当于不同的元素默认样式基本一样，然而一些浏览器自带的默认样式还是可以节省我们的时间的。

Normalize.css : http://necolas.github.io/normalize.css/

Web 安全字体

占据页面主导内容元素的大多是文字，Web 字体是由font-family指定的，可以同时指定多个，浏览器会自动挨个解析，保证前面的解析失败时，后面的可以备用，保证 Web 字体使用的安全性。 

<style>
	html, body { font-family: Helvetica, Tahoma, Arial, "Microsoft YaHei", "华文细黑", sans-serif; }
<style/>

这样的设置，兼具了 Windows 与 Mac OS 以及移动端设备，保证在任何操作系统与设备上都不会导致字体解析失败。通常将此属性放在公有 css 样式表中，最好加到body或html标签上，以保证其它元素的默认字体统一。需要注意的是，为何在这里强调 Web 字体，我们所熟知的“宋体”属于打印字体，也就是纸质打印时它是表现较好的，而 Web 展现上“宋体”应该避免，视觉效果并不好。

与字体相关的其实还有符号，为了实现符号的自适应，通常符号我们使用其编码。

HTML ISO-8859-1 参考手册 : http://www.runoob.com/tags/ref-entities.html

CSS 3 还引入了新的特性@font-face，使我们可以加载已有的字体资源文件，而彻底解决用户客户端系统没有安装所需要字体的问题。

Font Awesome : http://fontawesome.io/

Font Awesome 是一个字体和常用标志符号库，采用矢量符号图，可以满足我们的自适应需求。

CSS 布局核心知识

自从 DIV + CSS 这种页面布局开始成为主流，更考验我们开发者掌握的 CSS 布局核心知识，较为重要的有：

  • 浮动与清除浮动
  • 绝对与相对定位
  • 水平与垂直居中
  • z-index

通常来说，写一个 DOM 结构和样式时，我们应该在写完全部的 DOM 结构后，再去开始加 CSS 样式进行布局。而我们要养成的习惯是，先从子元素开始加样式、定宽高，这是极为重要的。很多人喜欢先定父元素宽高然后把子元素放进去，这是不对的，父元素应该遵循让子元素去自动撑开的原则。这样做的好处是，子元素与父元素结构紧密，整个页面结构更趋向于我们所期望的那样，也更方便维护。当页面结构出现问题，而你在不断更改多个元素宽高时，就要考虑考虑这种情况了。

元素的尺寸除了宽高以外，还有border、padding、margin，在 CSS3 中提供了新的样式属性box-sizing，其可取的值有border-box、content-box，前者定义元素的宽高包含元素内部、内边距和边框，而后者定义元素的宽高仅仅包含元素的内容，我们通常在写页面时将其统一一下，建议选择border-box取值。其次，扩充元素范围用padding，设置元素之间距离则用margin，一定要遵循这个原则，因为hover事件触发条件的范围包含元素的padding。

浮动是非常方便的一个特性，我们基本经常都会用到，而浮动起来简单，但浮动的子元素撑不起来父元素的高度是个很关键的副作用，一定要合理解决，合理解决不是说去给父元素设置一个定高，依然要遵循父元素让子元素撑开的原则，清除浮动的方法有很多，多了解多使用。

其次，页面不可能总是流式布局，当使用到position样式属性时，要清楚的知道该元素将会相对于哪个父元素进行偏移，很多时候随便设置了一个top、left效果实现了，结果换个分辨率、浏览器又不行了，这样的做法未免太过粗糙。

居中其实很多时候是一个硬性需求，毕竟视觉效果好，水平居中可能容易实现一些，而垂直居中就要难一些了，对于现在来说，是不推荐使用display的table值来实现子元素的垂直居中，让我们快速的将table标签有关的一切都忘记吧，使用一些技巧比如line-height和 CSS3 新特性实现垂直居中还是很简单的。

z-index 其实是一个很有用的样式属性，没了解过的人在调试某些问题时可能怎么也想不出原因，叠层元素显示优先级就靠它来指定。

我们所写的 CSS 样式是要重用的，不能一个元素用一套，那样工作量太大，效率太低，多使用.className，少使用#ID；少用行内样式，多用link引用样式表。

资源文件引入顺序

资源文件的引入顺序对于页面性能和显示效果来说非常重要，页面的加载是自上而下的，而且script脚本会阻塞页面渲染，link样式引入会导致页面重绘。所以，样式表文件应该全部在head里面引入，否则 DOM 结构加载完成后引入样式表会导致页面发生重绘；js 脚本文件应该在body末尾引入，这样可以避免 DOM 结构还没解析时找不到 DOM 元素的错误，也能让页面结构加载更快一些。

其次，对于同类资源文件，例如 CSS 文件，应先引入框架与库的样式文件，然后引入自己公用的样式文件，最后引入自己特定页面的样式文件。这样才会让我们自己所写的样式符合期望的应用到 DOM 元素上去，出了问题也更容易分析出原因。对于在特定页面中少量的需要覆盖公共样式的特定样式建议直接写在页面head的style标签内即可，js 也一样。

协作开发

通常一个项目是不可能单单由一个人来完成的，多人协作一般会使用项目代码管理工具 SVN、Git 来保证多人同时操作同一个文件时不发生冲突。然而，更重要的是每个人自己所干的事情如何让其它人更容易理解，提升交流的效率，首先就是要多写注释，写的方法、属性、样式让别人一眼就要能看出来是干什么的。其次就是如何解决命名冲突，写 CSS 样式时在进行 class 和 ID 命名时，建议加唯一标识符，例如自己名字缩写。

javaScript 的命名可能需要单独说明一下，这个语言本身没有像 Java、Ruby、PHP 那样鲜明的显式类结构，那么作用域一不小心就会变得混乱起来，直接定义的方法和变量将会挂载到全局对象身上，多人协作极易发生冲突，也不便于维护。那么，javaScript 中可以提供作用域的就是function了，在这里我们需要学会一个技巧。 

// 使用 Function 构建自己的作用域
function(){
	var i;
	function a(){}
	...
}

当然你会说这样写相当于一个匿名方法，没有调用的话，内部代码是不执行的，而写在全局作用域的代码是会自动执行的，接下来就要学会另一个技巧，自执行函数。 

// 使用自执行函数，将定义作用域中代码自动执行
!function(){
	var i;
	function a(){}
	...
}()

在方法前面加一个运算符，例如!，然后在方法体末尾加个括号()就会让该匿名方法定义时自动调用。然而，我们如何将自定义作用域中的一些属性和方法暴露给全局，便于其它作用域进行访问和引用呢？在括号中传入全局对象即可，其实还可以传全局作用域中其它的参数（属性、方法）。 

// 使用自执行函数，并将部分方法属性暴露给全局
!function(window){
	var i;
	function a(){}
	...

	// 暴露给全局的接口
	window.i = i;
	window.a = a;
}(window)

服务器调试

如果仅仅做的是前端页面，事实上不用装 Tomcat 这些看起来配置较为复杂的服务器，使用 http-server 这个零配置的 HTTP 服务器插件即可。页面其实也可以直接在本地打开预览，但对于整个项目来说，在服务器上调试可以更接近用户环境，容易发现使用中的诸多问题。

http-server : https://www.npmjs.com/package/http-server

使用 npm 即可安装：

npm install http-server [-g]

安装完成后，打开命令行工具，切换到项目根目录，输入以下命令即可：

http-server

SEO 与 SPA

SEO 即 Search Engine Optimization（搜索引擎优化），写前端页面事实上不是我们全部的工作，在页面实现过程中，应该根据项目考虑一下页面结构对 SEO 是否友好。当然，其实有专门做 SEO 的网站优化人员。通常来说，注重内容的网站对 SEO 的需求比较高，比如门户网站，个人博客等等；而像注重服务的网站应用则不太注重 SEO 排名，比如网页微信、网页QQ、管理页面。

SPA 即 Single Page Web Application（单页面应用程序），在一个主页面中异步加载整个应用的所有服务页面，传统的采用#也就是location.hash配合hashchange()事件的解决方案，目前还可以采用 HTML5 的新 API popState()和pushState()的解决方案。 SPA 带来的 Web 应用性能提升和流畅体验是显著的，堪比客户端软件，但 SPA 是对 SEO 不友好的。

SEO 与 SPA 放在一块说是比较合适的，因为像这种 SPA 技术以及目前的前端组件化开发框架，基本都采用动态加载 DOM 技术，然而对于搜索引擎来说只检索静态页面，不执行页面中的 js，那么动态加载的 DOM 就无法被搜索引擎检索到，并不利于 SEO 排名。

SSR

SPA 单页面应用与 SEO 的冲突其实有一种解决方案，SSR （Server Side Render）服务器端渲染。当然，SSR 不仅仅是为了解决这个问题而诞生的，像 WebQQ、新浪微博这种 Web App，首屏加载速度对用户体验来说是很重要的，服务器端渲染则可以加快首屏的加载速度。

通常的前端框架采用模版式语法动态生成 DOM 不利于 SEO，SSR 帮我们所做的其实就是将页面（首屏）主要数据直接在服务器端或者中间层获取到并生成页面内容，在响应客户端请求时将这些在服务器端渲染好的页面内容发送给客户端大大提高了页面的加载速度，并解决了一部分 SEO 的问题。
